- not supporting themes: the text colour in the fragment editor is hard coded, users of a light theme won't be able to see it
- connection to our chrome extension requires python 3
- man bekommt dauernd Fehler für tags, bei einem tag ("sd") und einem Fragment mit diesem Tag:
	############
	[W] | [TreeItem | get tag]: Failed for TreeItem: sd
	############
	> Object (kann man aufklappen)
	> Object (kann man aufklappen)
	[W] | [Database | getTreeItem]: Failed for label: sd
- alle 5 Sekunden wird überprüft ob die Datenbank sich geändert hat (ob aus der Chrome Erweiterung ein neues Fragment hinzugefügt wurde),
  dann wird die Listenansicht der Datenbank in der VSCode Erweiterung neu aufgebaut. Außerdem wird vor jedem schreiben in die Datenbank die 
  in memory Kopie komplett neu aufgebaut um versehentliches Überschreiben von neuen Fragmenten zu vermeiden
- Zum Testen beim Installieren mussten wir die Erweiterung packagen, deshalb ist "publisher" in der package.json gesetzt auf einen von uns kreierten Publisher.
- database sollte in globalStoragePath liegen, weil extensionPath bei extension update gewiped wird und somit die Datenbank weg ist.
- die Extension braucht eine Funktion zum Exportieren der Datenbank. Einem Nutzer ist normalerweise nicht bekannt in welchem Pfad die liegt und sie wird beim deinstallieren gelöscht
- Damit der native messaging host installiert wird, darf in out/frag.extract.host nicht extract.py vorhanden sein (das ist nur zum debuggen wichtig, vsce package ignoriert extract.py)
- Packagen nur in Linux oder auch auf Windows, wenn ihr die line ending Einstellungen in dem Repo ändert. Wird das in Windows gepackaged haben die Datein crlf line endings und damit
  kommt Linux nicht klar (Windows hat umgekehrt kein Problem mit Linux lf line endings)
